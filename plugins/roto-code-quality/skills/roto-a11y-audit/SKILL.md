# 접근성 감사 (Accessibility Audit)

컴포넌트나 페이지의 접근성 이슈를 WCAG 2.1 AA 기준으로 검출합니다.

## 사용 방법

```text
/roto-a11y-audit                       # 변경된 파일 자동 감지
/roto-a11y-audit [파일 또는 디렉토리]    # 특정 대상 감사
/roto-a11y-audit --pr                  # 현재 PR의 변경 파일만 감사
```

## 워크플로우

### Step 1: 대상 파일 수집

- **인자 없음**: `git diff origin/main...HEAD --name-only | grep '\.tsx$'`
- **인자 있음**: 지정된 파일/디렉토리의 TSX 파일
- **`--pr`**: PR diff에서 변경된 TSX 파일

### Step 2: 정적 분석

각 파일을 읽어 아래 항목을 검사합니다:

#### 이미지 및 미디어
- `<img>` / `<Image>`에 `alt` 속성 존재 여부
- 장식용 이미지에 `alt=""` 또는 `aria-hidden="true"` 적용 여부
- `<video>`, `<audio>`에 자막/대체 텍스트 존재 여부

#### 인터랙티브 요소
- `<button>`, `<a>`에 접근 가능한 이름 (텍스트 또는 `aria-label`)
- `<input>`에 연결된 `<label>` 또는 `aria-label`
- `focus-visible` 스타일 존재 여부
- 클릭 핸들러가 있는 비인터랙티브 요소 (`<div onClick>`)
- 터치 대상 최소 크기 (44x44px)

#### 시맨틱 구조
- 헤딩 레벨 순서 (`h1` → `h2` → `h3`)
- 랜드마크 역할 (`<nav>`, `<main>`, `<aside>`)
- `<nav>`에 `aria-label` 존재 여부
- 리스트 구조 적절성 (`<ul>`, `<ol>`)

#### 색상 및 대비
- 텍스트 색상과 배경색의 대비율 계산 (WCAG AA: 4.5:1, AAA: 7:1)
- 하드코딩된 색상 값의 대비율 체크
- 색상만으로 정보를 전달하지 않는지 확인

#### ARIA 사용
- `aria-hidden="true"` 적용 대상의 적절성 (장식용 아이콘 등)
- `role` 속성의 올바른 사용
- `aria-live` 영역의 적절한 배치 (동적 콘텐츠)
- 불필요한 ARIA 사용 (네이티브 HTML로 충분한 경우)

#### 모션 및 애니메이션
- `prefers-reduced-motion` 미디어 쿼리 대응 여부
- 자동 재생 애니메이션에 정지 수단 존재 여부
- 깜빡임 빈도 (3Hz 이하 권장)

#### 폼
- 에러 메시지와 입력 필드의 연결 (`aria-describedby`)
- 필수 필드 표시 (`required`, `aria-required`)
- 폼 제출 후 포커스 관리

### Step 3: 보고서 생성

```markdown
## 접근성 감사 보고서

### 📊 요약
| 수준 | 개수 |
|------|------|
| 🔴 Critical (WCAG A 위반) | N |
| 🟠 Major (WCAG AA 위반) | N |
| 🟡 Minor (모범 사례 미준수) | N |
| 🔵 Info (권장사항) | N |

### 이슈 목록

| # | 수준 | 파일:라인 | WCAG | 제목 | 설명 |
|---|------|----------|------|------|------|
| 1 | 🔴 | Button.tsx:42 | 1.1.1 | 이미지에 alt 누락 | ... |
| 2 | 🟠 | Card.tsx:18 | 1.4.3 | 대비율 부족 (2.8:1) | ... |

### 파일별 상세

#### {파일명}
- 이슈 설명
- 수정 제안
- 참고: WCAG 가이드라인 링크
```
